# 招投标管理系统实施路线图

## 总体策略

- 先冻结核心边界和状态机，再并行开发
- 先做内部主系统闭环，再补轻量门户
- 先跑通固定流程，再决定是否引入更强流程引擎

## P0 设计定版

### 范围

- 角色矩阵
- 核心状态机
- 核心数据模型
- 菜单与权限码
- API 契约
- 附件归属与版本规则

### 验收标准

- 关键状态图评审通过
- 前后端命名统一
- 核心表与主接口不再频繁变动
- 关键权限码与角色矩阵冻结

### 风险

- 业务边界失控
- 过早追求 BPM 化
- 评标规则未收口

## P1 MVP-1

### 范围

- 项目 / 标段管理
- 招标文件与公告
- 供应商主数据关联
- 报名 / 邀请
- 投标登记与附件
- 门户基础能力

### 验收标准

- 能从建项目走到“待开标”
- 三类主体都能登录并完成各自基础动作
- 附件、回执、日志、消息可用
- 至少 3 类角色权限可验证

### 风险

- 供应商数据重复
- 招标文件版本混乱
- 时间节点控制复杂

## P2 MVP-2

### 范围

- 开标记录
- 评标模板与评分
- 结果汇总
- 定标确认
- 归档与导出

### 验收标准

- 能从“待开标”走到“已定标”
- 评分记录可追溯
- 定标结果可导出
- 归档包可按项目和标段回查

### 风险

- 评分规则变化
- 专家权限隔离不清
- 并发修改冲突

## P3 增强版

### 候选范围

- 供应商门户增强
- CA 签章
- 加解密开标
- 监管平台对接
- 更丰富统计分析
- 移动端或消息增强

### 验收标准

- 小范围试点稳定
- 外部依赖联调通过
- 合规性评审通过

### 风险

- 外部依赖多
- 合规和安全成本上升
- 范围快速膨胀

## 并行开发建议

- 串行冻结
  - 核心实体
  - 状态机
  - API 契约
  - 权限码
  - 主表设计
- 可并行流
  - 流 1：项目 / 标段 / 招标文件
  - 流 2：供应商 / 报名 / 投标提交
  - 流 3：开标 / 评标 / 定标
  - 流 4：附件 / 消息 / 导出 / 日志
  - 流 5：前端页面壳、Mock、联调

## 质量门槛

- 后端最小回归
  - 登录
  - 权限
  - 附件上传
  - 核心动作接口
- 前端最小回归
  - 动态菜单
  - 主列表和表单
  - 门户投标提交流程
- 端到端回归
  - 建项目 -> 发布 -> 报名 -> 投标 -> 开标 -> 评标 -> 定标 -> 归档
