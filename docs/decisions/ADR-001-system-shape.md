# ADR-001: 沿用现有单体后端 + 独立前端形态

## Status

Accepted

## Date

2026-06-07

## Context

当前仓库已经具备：

- 后端单体基础能力
  - 权限、菜单、组织、文件、消息、日志、字典
- 前端管理端基础能力
  - 登录、权限、动态菜单、请求封装、列表表单页框架

招投标管理系统虽然业务复杂，但一期目标是可控落地，而不是架构重做。

## Decision

继续采用现有单体后端 `sa-base + sa-admin` 和独立 Vue 管理端形态，新增 `bid` 业务域。

注：首批实现已按 `module/system/bid` 落地，业务域物理路径由 `ADR-004-bid-p0-implementation-baseline.md` 修订；本 ADR 的单体形态决策继续有效。

## Alternatives Considered

### 先拆微服务

- Pros
  - 未来边界更清晰
- Cons
  - 一期成本高
  - 基础设施和联调成本暴涨
  - 会把当前重点从业务落地转移到架构治理

### 重新搭建新仓库

- Pros
  - 代码更整洁
- Cons
  - 无法复用当前成熟底座
  - 交付周期更长

## Consequences

- 一期可以快速复用现有能力
- 后续如果业务体量显著增加，再评估服务拆分
- 招投标业务必须保持清晰业务域边界，避免在单体内无序扩散
